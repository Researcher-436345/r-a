# Roadmap продукта Researcher

## 1. Цель продукта

Создать единую библиотеку научных статей, в которой пользователь:

- хранит PDF и библиографические карточки;
- импортирует материалы из разных источников;
- организует статьи по коллекциям и тегам;
- быстро находит сохраненные материалы;
- читает статьи и сохраняет заметки;
- формирует оформленный список литературы;
- использует AI для краткого содержания, объяснений и вопросов по тексту.

Первая версия должна доказать основной сценарий:

> Добавить статью один раз, сохранить ее вместе с корректной библиографией и быстро найти позже.

## 3. Модель данных

### Глобальные сущности

`papers`

- внутренний идентификатор;
- название;
- abstract;
- год публикации;
- журнал или конференция;
- DOI;
- arXiv ID;
- язык;
- тип публикации;
- дата создания и обновления.

`paper_versions`

- статья;
- номер версии;
- источник;
- ссылка на оригинал;
- ключ PDF в object storage;
- SHA-256 файла;
- размер файла;
- статус обработки.

`authors`

- имя;
- нормализованное имя;
- ORCID, если известен.

`paper_authors`

- статья;
- автор;
- порядок автора.

`references`

- статья, из которой взята ссылка;
- найденная целевая статья;
- исходный текст ссылки;
- DOI или другой идентификатор;
- уверенность сопоставления.

### Пользовательские сущности

`user_library_items`

- пользователь;
- статья;
- дата добавления;
- статус: unread, reading, read;
- favorite;
- личная заметка.

`collections`

- пользователь;
- название;
- описание.

`collection_items`

- коллекция;
- статья;
- порядок внутри коллекции.

`tags` и `library_item_tags`

- пользовательские теги и их связи со статьями.

`annotations`

- статья;
- пользователь;
- страница;
- координаты выделения;
- выделенный текст;
- заметка;
- цвет.

## 4. Архитектура хранения

### Компоненты

- PostgreSQL — пользователи, статьи, библиография, коллекции и служебные данные;
- S3-совместимое object storage — оригинальные PDF;
- Redis — очередь задач и временный кеш;
- worker — скачивание и обработка PDF;
- GROBID — извлечение структуры статьи и библиографии;
- Crossref и OpenAlex — получение и проверка метаданных;
- PostgreSQL Full Text Search, `pg_trgm` и `pgvector` — поиск;
- LLM API — summary, explain и чат;
- CDN — быстрая доставка PDF.

### Правила хранения

1. Публичная статья хранится в одном экземпляре для всех пользователей.
2. Загруженные пользователем закрытые материалы являются приватными.
3. Одинаковые файлы определяются по SHA-256.
4. DOI и arXiv ID являются основными ключами дедупликации.
5. Каждая версия статьи хранится отдельно, если содержимое изменилось.
6. Удаление статьи из личной библиотеки удаляет связь пользователя, но не обязательно общий PDF.
7. Физическое удаление файла выполняется, когда на него больше никто не ссылается и истек срок хранения.
8. Доступ к приватным PDF выдается через короткоживущие signed URLs.

### Путь добавления статьи

```text
PDF / DOI / arXiv / BibTeX
              ↓
Нормализация идентификатора
              ↓
Поиск существующей статьи
              ↓
Получение или сохранение PDF
              ↓
Извлечение метаданных и references
              ↓
Проверка через Crossref/OpenAlex
              ↓
Создание поискового индекса
              ↓
Добавление в библиотеку пользователя
```

Карточка должна появляться сразу. Долгие этапы выполняются асинхронно со статусами:

- файл загружен;
- извлекается текст;
- обрабатывается библиография;
- строится поисковый индекс;
- готово.

## 5. Поиск

### Поисковые поля и веса

1. точный DOI, arXiv ID или другой идентификатор;
2. название статьи;
3. авторы;
4. название журнала или конференции;
5. год и ключевые слова;
6. abstract;
7. полный текст;
8. пользовательские заметки и теги.

Название и авторы должны иметь больший вес, чем совпадение в полном тексте.

### Технология первой версии

Использовать PostgreSQL:

- B-tree/unique indexes для DOI и arXiv ID;
- Full Text Search для названия, abstract и текста;
- `pg_trgm` для опечаток и неточных названий;
- `pgvector` для смыслового поиска.

OpenSearch или Elasticsearch добавлять только после измеримого ухудшения скорости или качества на реальном объеме.

### Фильтры

- автор;
- год;
- коллекция;
- тег;
- статус чтения;
- источник;
- наличие PDF;
- журнал или конференция;
- дата добавления.

### Поиск по библиографии

Для каждой распознанной ссылки:

1. искать точный DOI;
2. искать arXiv ID;
3. сопоставлять нормализованные название, год и первого автора;
4. применять fuzzy matching;
5. сохранять исходную ссылку, если статья не определена.

Пользовательские сценарии:

- найти статью из списка литературы в своей библиотеке;
- открыть все источники текущей статьи;
- добавить выбранные источники в коллекцию;
- показать, какие статьи в библиотеке ссылаются друг на друга;
- найти дубликаты;
- сформировать библиографию по коллекции.

## 6. Библиография в MVP

### Импорт

- DOI;
- arXiv ID и URL;
- BibTeX;
- RIS;
- PDF;
- ручное создание записи.

### Автоматически заполняемые поля

- название;
- авторы и их порядок;
- год;
- journal/conference;
- volume;
- issue;
- pages;
- publisher;
- DOI;
- arXiv ID;
- URL;
- дата обращения для web-источников.

### Редактирование

Пользователь должен иметь возможность исправить любое поле. Автоматически полученные данные нельзя считать безошибочными.

### Экспорт

- одна статья;
- выбранные статьи;
- вся коллекция;
- вся библиотека.

Форматы:

- BibTeX;
- RIS;
- CSL-JSON;
- plain text;
- копирование в буфер обмена.

### Оформление

Использовать готовый CSL-движок, например `citeproc-js`.

Первая поставка стилей:

- ГОСТ;
- APA;
- IEEE.

Позднее пользователь сможет выбирать стили из каталога Citation Style Language.

## 7. Этапы разработки

### Этап 0. Технический прототип — 1–2 недели

Цель: проверить критические технические гипотезы.

- загрузить PDF в object storage;
- получить metadata по DOI и arXiv;
- распарсить 50–100 разных PDF через GROBID;
- проверить качество извлечения русских и английских references;
- проверить дедупликацию по DOI, arXiv ID и SHA-256;
- собрать прототип поиска в PostgreSQL;
- измерить время и стоимость обработки одного PDF.

Критерий завершения:

- не менее 90% цифровых PDF успешно обрабатываются;
- DOI и название корректно определяются для большинства тестового набора;
- одинаковые статьи не создают дубликаты.

### Этап 1. Ядро библиотеки — 3–4 недели

- регистрация и авторизация;
- `papers`, `paper_versions`, `authors`;
- загрузка PDF;
- добавление DOI и arXiv;
- object storage;
- личная библиотека;
- карточка статьи;
- коллекции и теги;
- статусы обработки;
- базовая дедупликация.

Критерий завершения:

Пользователь может добавить статью, увидеть ее в библиотеке и повторно открыть PDF с другого устройства.

### Этап 2. Библиография и поиск — 3–4 недели

- импорт BibTeX и RIS;
- редактирование метаданных;
- экспорт BibTeX, RIS и CSL-JSON;
- ГОСТ, APA и IEEE;
- генерация библиографии по коллекции;
- поиск по названию, авторам, DOI и abstract;
- фильтры;
- извлечение references;
- поиск дубликатов.

Критерий завершения:

Пользователь импортирует набор статей, находит нужную работу и получает оформленный список литературы.

### Этап 3. Чтение и AI — 3–4 недели

- web PDF-ридер;
- выделения и заметки;
- автоматическое краткое содержание;
- explain по выделенному фрагменту;
- чат по одной статье;
- ссылки ответа на страницы и фрагменты PDF;
- лимиты использования.

Критерий завершения:

Ответы AI опираются на конкретные фрагменты документа, а пользователь может открыть использованную страницу.

### Этап 4. Связи и discovery — 3–5 недель

- похожие статьи;
- переходы по списку литературы;
- внутренний граф связей библиотеки;
- добавление источников в один клик;
- обзор литературы по выбранной коллекции;
- рекомендации на основе библиотеки.

Критерий завершения:

Пользователь может перейти от одной статьи к связанным работам и получить обзор выбранного набора.

### Этап 5. Beta и масштабирование — 2–4 недели

- импорт из Zotero;
- массовая загрузка;
- адаптивный интерфейс для телефона;
- наблюдаемость и аналитика;
- резервное копирование;
- управление квотами;
- удаление и экспорт пользовательских данных;
- нагрузочное и security-тестирование;
- тарифы Free и Pro.

## 8. Приоритеты при нехватке времени

Порядок, который нельзя нарушать:

1. надежное сохранение PDF;
2. отсутствие дублей;
3. корректная библиографическая карточка;
4. поиск;
5. экспорт и оформление библиографии;
6. удобное чтение;
7. AI-функции;
8. рекомендации и граф.

Если сроки сокращаются, следует уменьшать количество AI-сценариев или интеграций, но не исключать библиографию.

## 9. Метрики первой версии

### Activation

- пользователь добавил первую статью;
- пользователь добавил не менее пяти статей;
- хотя бы одна статья получила корректную библиографическую карточку;
- пользователь создал коллекцию или экспортировал библиографию.

### Основная ценность

- число сохраненных статей на активного пользователя;
- доля повторно открытых статей;
- количество поисков;
- доля успешных поисков без повторного запроса;
- количество экспортов библиографии;
- количество коллекций с пятью и более статьями.

### Качество обработки

- доля успешно обработанных PDF;
- доля metadata без ручного исправления;
- доля найденных DOI;
- доля корректно сопоставленных references;
- количество созданных дублей;
- время от загрузки до готовности карточки.

### AI

- доля пользователей summary, explain и chat;
- стоимость AI на активного пользователя;
- доля ответов, после которых открыли указанный фрагмент;
- количество отрицательных оценок ответа.

### Retention

- возврат через 7 и 30 дней;
- повторное открытие сохраненных материалов;
- прирост библиотеки в месяц.

## 10. Ключевые технические риски

### Ошибки метаданных

PDF может не содержать DOI или иметь нестандартную верстку.

Меры:

- объединять данные GROBID, Crossref, OpenAlex и arXiv;
- показывать источник каждого поля;
- разрешать ручное редактирование.

### Дубликаты

Одна статья может иметь arXiv-версию, DOI и загруженный PDF.

Меры:

- таблица всех внешних идентификаторов;
- SHA-256;
- fuzzy matching названий;
- ручное объединение записей администратором или пользователем.

### Лицензии

Не все PDF разрешено хранить как общедоступные.

Меры:

- разделять открытые общие и приватные пользовательские файлы;
- проверять источник и лицензию;
- не публиковать пользовательские загрузки;
- поддержать удаление по запросу правообладателя.

### Приватность

Пользователь может загрузить закрытую рукопись.

Меры:

- приватность по умолчанию;
- signed URLs;
- проверка прав на каждом запросе;
- изоляция поиска по `user_id`;
- запрет использования приватных документов для обучения моделей без отдельного согласия.

### Качество PDF

Сканированные документы требуют OCR, а сложные формулы и таблицы могут распознаваться плохо.

Меры:

- в MVP поддержать прежде всего цифровые PDF;
- помечать документы с низким качеством извлечения;
- добавить OCR отдельным этапом после MVP.

## 11. Решения, которые нужно принять до разработки

1. Какой S3-провайдер и регион хранения использовать.
2. Какие страны и требования к персональным данным являются целевыми.
3. Считать ли загруженные PDF приватными всегда.
4. Какие конкретные варианты ГОСТ нужны первой аудитории.
5. Какой стек используется для backend и frontend.
6. Будет ли продукт хранить несколько версий одной статьи.
7. Какие лимиты библиотеки и AI будут у бесплатного тарифа.
8. Нужно ли разрешить полный экспорт библиотеки вместе с PDF.

## 12. Предлагаемый стек для быстрого MVP

Один из практичных вариантов:

- frontend — Next.js;
- backend API — FastAPI или NestJS;
- PostgreSQL;
- `pgvector` и `pg_trgm`;
- S3/Cloudflare R2/Yandex Object Storage;
- Redis;
- Celery для Python или BullMQ для Node.js;
- GROBID как отдельный Docker-сервис;
- PDF.js для ридера;
- Crossref и OpenAlex для метаданных;
- `citeproc-js` для оформления библиографии;
- внешний LLM API для summary, explain и chat.

Конкретный выбор FastAPI или NestJS следует делать по компетенциям команды, а не по предполагаемой производительности: на масштабе MVP оба варианта достаточны.

## 13. Бэклог идей (после итерации 1+)

Идеи зафиксированы, но **не входят** в текущий MVP. Брать в работу только после закрытия P0/P1 итерации 1.

### Discovery: гибридная лента рекомендаций

**Статус:** идея / backlog  
**Когда:** итерация 2–3+ (после простой trending-ленты с arXiv)

**Суть:** сложная персональная лента, где к «холодным» трендам из внешних источников подмешиваются сигналы поведения пользователей нашей платформы.

**Возможные сигналы:**
- что чаще **добавляют в библиотеку** (по категориям / arXiv id / авторам);
- что ищут в Ask / web-search;
- статусы чтения и favorite;
- пересечения интересов между похожими пользователями (collaborative filtering light).

**Источники для подмешивания:**
- arXiv trending / свежие (уже в итерации 1);
- позже — OpenAlex, Semantic Scholar, Hugging Face papers;
- внутриплатформенный score популярности статей в Researcher.

**Набросок пайплайна:**
1. Базовый слой: свежие / trending из внешних API (кеш).
2. Слой платформы: топ добавляемых / читаемых за период.
3. Персональный слой: boost статей рядом с уже сохранёнными (категория, авторы, эмбеддинги abstract).
4. Смешивание и дедуп → `GET /feed/for-you`.

**Не делать слишком рано:** нужна критическая масса пользователей и логов событий (search, add_to_library, open_reader), иначе лента будет шумной.

### Свободные заметки (без привязки к выделению)

**Статус:** идея / backlog  
**Когда:** после EPIC-06 (аннотации на выделениех уже есть)

**Суть:** сейчас заметка почти всегда рождается из выделения в PDF. Нужны «обычные» заметки — текст без quote/rect.

**Варианты модели (выбрать один или комбинировать):**

1. **Заметка на статью (paper note)**  
   Одна или несколько свободных заметок `paper_id + user_id`, без `page/rect/selected_text`. Подходит для «итоги по статье», TODO, вопросов к семинару.

2. **Одна большая заметка на статью**  
   Как Notion-страница / scratchpad: один документ на `paper_id`, автосейв. Проще UX, хуже история версий.

3. **Заметки на проект**  
   Когда появятся проекты (EPIC-05): заметка уровня `project_id` — обзор литературы, план чтения, выводы по группе статей.

4. **Глобальные / кросс-статейные заметки**

**Не путать с:** annotations (highlight + quote) и chat_messages (диалог с AI).
